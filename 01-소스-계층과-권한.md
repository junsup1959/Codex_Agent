# 소스 계층과 권한 모델

> [!summary]
> 이 문서는 어떤 파일이 어떤 결정을 내리는지 설명한다. 현재 구조의 핵심은 “문서, skill, contract, validator, MCP ledger가 각자 다른 권한을 가진다”는 점이다.

## 최상위 권한 계층

| 계층 | 경로 | 권한 |
| --- | --- | --- |
| 전역 포인터 | `${CODEX_HOME}\AGENTS.md` | 모든 작업에 적용되는 compact guardrail |
| 아키텍처 인덱스 | `${CODEX_HOME}\agent-architecture\AGENT-ARCHITECTURE.md` | 상세 문서의 위치와 읽는 순서 |
| 아키텍처 매퍼 | `${CODEX_HOME}\agent-architecture\AGENT-ARCHITECTURE-MAPPER.md` | `/init`, repo-local `AGENTS.md`, runtime 문서 연결 |
| 런타임 절차 | `${CODEX_HOME}\agent-architecture\09-runtime-orchestration-steps.md` | architecture-required 작업의 stage 순서 |
| 상세 계약 | `${CODEX_HOME}\agent-architecture\00-*.md` ~ `09-*.md` | stage별 artifact, gate, failure class |
| 필수 skill | `${CODEX_HOME}\skills\<skill>\SKILL.md` | main agent가 실제로 따르는 절차 |
| skill contract | `${CODEX_HOME}\skills\<skill>\contract.json` | validator가 읽는 machine-readable stage contract |
| agent catalog | `${CODEX_HOME}\agents\<category>\*.toml` | 실제 spawn 가능한 specialist role 목록 |
| run context | `codex-context-ledger` MCP | run 단위 context packet, artifact refs, stage pass refs |
| validator | `${CODEX_HOME}\agent-architecture\*.py` | 문서/contract/runtime artifact가 규칙을 지키는지 판단 |

## `AGENTS.md`의 역할

`AGENTS.md`는 길어지면 안 된다. 이 파일은 다음 세 가지를 빠르게 알려주는 pointer다.

1. 무엇을 먼저 읽어야 하는지
2. 언제 `architecture_required=true`가 되는지
3. 어떤 global guardrail을 절대 어기면 안 되는지

따라서 세부 규칙을 `AGENTS.md`에 계속 추가하면 안 된다. 세부 규칙은 `agent-architecture/*.md`, skill `SKILL.md`, `contract.json`, validator로 내려야 한다.

## Skill의 권한

Skill은 “어떻게 실행할지”를 설명한다. 예를 들어 `$worker-router`는 router stage에서 어떤 입력을 읽고, specialist list를 어떻게 소환하며, 무엇을 output으로 내보낼지 설명한다.

하지만 skill이 validator를 대체하지 않는다. Skill은 절차이고, validator는 기계 검증이다. 그래서 현재 구조는 skill 옆에 `contract.json`을 두고, skill 실행 전 `Contract Gate`에서 contract를 읽도록 했다.

## `contract.json`의 권한

`contract.json`은 다음을 기계적으로 명시한다.

| 필드 | 의미 |
| --- | --- |
| `schema_version` | contract 형식 버전 |
| `skill_name` | skill 이름과 폴더 이름이 일치해야 함 |
| `activation_ref` | `$orchestrator` 같은 활성화 이름 |
| `mandatory_for_architecture_required` | architecture-required일 때 필수인지 여부 |
| `stage_owner` | 논리 stage owner |
| `stage_execution_mode` | 기본 실행 방식. 대부분 `main_agent_role_pass` |
| `purpose` | 이 stage가 하는 일 |
| `input_artifacts` | 반드시 읽어야 할 입력 artifact |
| `output_artifacts` | 내보낼 수 있는 canonical output |
| `forbidden_outputs` | 이 stage에서 절대 만들면 안 되는 output |
| `required_evidence` | handoff 전에 필요한 evidence |
| `source_docs` | 참조해야 할 architecture MD |

이 구조 때문에 validator는 skill 내용을 전부 자연어로 해석하지 않아도 된다. 먼저 contract shape와 필수/선택 경계를 확인하고, full validator에서 문서와 prompt까지 함께 본다.

## Agent catalog의 권한

실제 spawn 가능한 role은 `${CODEX_HOME}\agents\<category>\*.toml`에서 나온다. Router가 memory나 감으로 role을 만들면 안 된다.

특히 `$worker-router`, `$review-router`는 반드시 specialist list를 다시 소환해야 한다. “소환”은 모든 agent를 spawn한다는 뜻이 아니라, catalog를 enumeration해서 가능한 concrete role 목록을 확보한다는 뜻이다.

## Obsidian 문서의 권한

이 폴더의 문서는 개인 기록이다. 여기 있는 설명이 실제 runtime을 바꾸지 않는다.

Obsidian 문서는 다음에 쓰인다.

- 구조를 설명하기 위한 장기 메모
- 장애 대응 시 빠른 탐색
- 아키텍처 의사결정 기록
- validator와 skill contract의 관계를 사람이 이해하기 위한 문서

Obsidian 문서가 오래되면 `.codex` 원본을 기준으로 갱신해야 한다.
