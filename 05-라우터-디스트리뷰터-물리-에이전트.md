# 라우터, 디스트리뷰터, 물리 에이전트

> [!summary]
> router는 논리 manifest만 만든다. distributor가 실제 child agent를 materialize하고 wait evidence를 기록한다.

## 왜 분리했나

예전 구조에서 router가 role 선택과 physical spawn을 동시에 맡으면 책임이 섞인다. 이러면 다음 문제가 생긴다.

- launch manifest에 실제 `agent_id`나 `wait_handle`이 섞인다.
- logical routing 실패와 physical spawn 실패가 구분되지 않는다.
- aggregation 시 어떤 lane이 실제로 spawn되었는지 추적하기 어렵다.
- thread limit, timeout, schema invalid 같은 failure class가 흐려진다.

현재 구조는 이를 분리한다.

```text
task-planner -> worker-router -> task-distributor -> physical worker specialists
aggregator -> review-router -> task-distributor -> physical reviewer specialists
```

## Router 책임

Router의 책임은 다음이다.

1. 현재 plan 또는 aggregation packet을 읽는다.
2. specialist catalog를 enumerate한다.
3. fanout budget을 고려해 최소 lane을 선택한다.
4. logical `launch_manifest`를 만든다.
5. schema gate를 통과할 수 있는지 확인한다.

Router가 하면 안 되는 것:

- `spawn_agent`
- `wait_agent`
- `close_agent`
- `active_passes` 생성
- `agent_id` 생성
- `submission_id` 생성
- generic `worker` role 선택
- skill 이름을 `agent_role`로 사용

## Specialist List Gate

`$worker-router`와 `$review-router`는 모두 specialist list gate가 있다.

정확한 의미:

- `${CODEX_HOME}\agents\<category>\*.toml`을 열거한다.
- 실제 존재하는 TOML role만 spawn 가능한 role로 본다.
- memory, family alias, skill name, partial list로 routing하지 않는다.
- “모든 specialist 사용”은 catalog 전체를 평가하라는 뜻이지 전부 spawn하라는 뜻이 아니다.

## Launch Manifest

`launch_manifest`는 논리 계획이다. 예시 필드:

- lane id
- `agent_category`
- `agent_role`
- ownership
- expected artifact
- validation prompt
- merge point
- `context_packet_version`
- `caller_spawn_required=true`
- `initial_status=unmaterialized`

`launch_manifest.children[]`에는 실제 wait handle이 있으면 안 된다. wait handle은 physical spawn 이후 `active_passes`에 기록된다.

## Distributor 책임

`$task-distributor`는 validated manifest를 실제 실행으로 바꾼다.

해야 할 일:

1. manifest가 schema-valid인지 확인한다.
2. fanout budget을 확인한다.
3. budget 안에서만 next wave를 materialize한다.
4. concrete TOML-backed specialist role로만 `spawn_agent`한다.
5. `spawn_receipt_ref`, `agent_id`, `submission_id`, `wait_handle`을 기록한다.
6. 모든 materialized child에 대해 `wait_agent`를 호출한다.
7. lane status를 returned, superseded, timed out, schema-invalid, blocked, failed 등으로 분류한다.
8. `active_passes`와 `handoff_result` evidence를 다음 stage에 넘긴다.

## Fanout Budget

기본 fanout budget은 보수적이다.

| 제한 | 기본값 |
| --- | --- |
| worker lane per execution wave | 2 |
| reviewer lane per review wave | 2 |
| total child agents per loop | 4 |
| same-role parallel lane | 1 |
| MCP-using child lane active at once | 1 |

budget을 넘는 경우:

- lane을 합친다.
- 같은 role을 병렬로 여러 개 만들지 않는다.
- 넘친 review axis는 waiver 또는 feedback으로 명시한다.
- over-spawning으로 해결하지 않는다.

## Worker Specialist

worker specialist는 실제 산출물을 만든다. 예:

- 코드 수정
- 테스트 추가
- 문서 작성
- 데이터 분석
- 인프라 검토
- 디버깅

worker는 반드시 bounded scope를 받아야 한다. full conversation history에 기대면 안 된다. `spawn_context_mode="scoped_packet"`이 기본이다.

## Reviewer Specialist

reviewer specialist는 aggregation 결과를 검토한다. review axis는 보통 다음 중 하나다.

- correctness
- security
- performance
- accessibility
- architecture
- test coverage
- compliance

reviewer도 router manifest와 distributor materialization을 통해 생성된다. 직접 reviewer를 감으로 spawn하면 architecture claim이 깨진다.

## Aggregation 전 차단 조건

다음 중 하나라도 있으면 aggregation으로 진행하면 안 된다.

- spawn 실패가 unclassified 상태
- thread limit이 retry/waiver 없이 방치됨
- materialized child가 `wait_agent` 대상이 아님
- `close_agent`만 있고 wait evidence가 없음
- handoff result가 schema-invalid인데 정상 결과로 취급됨
- missing lane이 분류되지 않음

`close_agent`는 cleanup일 뿐 wait evidence가 아니다.
