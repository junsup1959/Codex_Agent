# 운영 플레이북

> [!summary]
> 이 문서는 현재 Agents 구조를 수정하거나 점검할 때의 실제 운영 순서를 정리한다.

## 구조 변경 전 확인

아키텍처, skill, validator, runtime prompt를 건드리기 전 확인할 것:

1. `${CODEX_HOME}\AGENTS.md`
2. `${CODEX_HOME}\agent-architecture\09-runtime-orchestration-steps.md`
3. 관련 detail doc: `00-*.md` ~ `08-*.md`
4. 관련 skill: `${CODEX_HOME}\skills\<skill>\SKILL.md`
5. 관련 contract: `${CODEX_HOME}\skills\<skill>\contract.json`
6. validator module: `${CODEX_HOME}\agent-architecture\validators\*.py`

## stage skill 수정 절차

예: `$worker-router`를 바꿀 때

1. `skills\worker-router\SKILL.md`를 읽는다.
2. `skills\worker-router\contract.json`을 읽는다.
3. `source_docs`에 지정된 architecture MD를 읽는다.
4. SKILL과 contract의 책임이 충돌하지 않도록 수정한다.
5. router가 spawn/wait하지 않는다는 경계를 유지한다.
6. `validate-skill-contracts.py`를 실행한다.
7. full validator를 실행한다.

명령:

```powershell
python ${CODEX_HOME}\agent-architecture\validate-skill-contracts.py
python ${CODEX_HOME}\agent-architecture\validate-agent-architecture.py
```

## runtime 절차 변경 절차

`09-runtime-orchestration-steps.md`를 바꾸면 다음도 확인한다.

- `AGENTS.md` pointer가 여전히 맞는지
- `AGENT-ARCHITECTURE.md`와 mapper가 새 순서를 가리키는지
- `validators/constants.py`의 required text가 갱신되어야 하는지
- `validate-runtime-gate-smoke.py` fixture가 새 실패 조건을 반영하는지
- skill `contract.json`의 `source_docs` 또는 required evidence가 바뀌는지

필수 검증:

```powershell
python ${CODEX_HOME}\agent-architecture\validate-agent-architecture.py
python ${CODEX_HOME}\agent-architecture\validate-runtime-gate-smoke.py
```

## 흔한 장애와 판정

| 증상 | 원인 | 처리 |
| --- | --- | --- |
| router가 직접 child를 spawn함 | router/distributor 책임 혼합 | router는 manifest만 내고 distributor로 돌린다 |
| launch manifest에 `agent_id`가 있음 | logical/physical ledger 혼합 | `agent_id`는 `active_passes`로 이동 |
| final claim이 빨리 나옴 | feedback gate evidence 부족 | `judgment_envelope`와 reviewer evidence 확인 |
| `close_agent`만 있음 | wait evidence 아님 | `wait_agent` evidence 또는 explicit classification 필요 |
| Docker MCP 호출 실패를 성공처럼 기록 | error-shaped MCP output 오판 | blocker/waiver evidence로 분류 |
| context가 대화 기억에만 있음 | context ledger 미사용 | `codex-context-ledger`에 context packet 기록 |
| 모든 specialist를 spawn하려 함 | “use all specialists” 의미 오해 | catalog 평가 후 최소 bounded set 선택 |

## 변경 후 체크리스트

- [ ] MCP cleanup snapshot이 요구되는가

## 현재 기준 검증 baseline

최근 확인된 baseline:

```text
validate-skill-contracts.py: PASS checks=199
validate-agent-architecture.py: PASS checks=1703
validate-runtime-gate-smoke.py: PASS cases=72
quick_validate.py: 8 skills valid
```

baseline 숫자는 validator scope가 넓어지면 증가할 수 있다. 숫자가 줄어들면 validator coverage가 빠졌을 가능성을 의심해야 한다.

## 기록 갱신 규칙

이 Obsidian 폴더는 runtime source가 아니므로, 먼저 `.codex` 원본을 고친 뒤 이 문서를 갱신한다.

문서 갱신 시:

1. 오래된 파일을 남겨 충돌시키지 않는다.
2. current architecture 기준으로 다시 쓴다.
3. 경로는 `${CODEX_HOME}` 기준으로 적는다.
4. 검증 수치는 실제 실행 결과를 적는다.
5. optional helper를 mandatory gate처럼 쓰지 않는다.
6. memory-derived 사실은 live verification과 구분한다.
